# Draw in the Air — Trust & Scale Audit

**Date:** 2026-06-13
**Scope:** Security, data/RLS, payments, scalability (10 → 1M users), code health, edge cases, and the tests to write before shipping.
**Method:** Read of all 20 Supabase migrations, all 5 Stripe/email edge functions, the client auth/data layer, and a cross-reference pass over ~292 source files (~71k LOC). Every finding below is cited to a file and line.

> **Bottom line:** The foundations are better than most apps at this stage. Privacy-by-design for kids is real (no photos/DOB/full names stored, consent gates, working deletion/export flows), RLS is competently tuned, there is **no SQL injection**, and **no Stripe or service-role key was ever committed**. The product's expensive part — camera hand-tracking — runs entirely in the browser, so it scales for free. What stands between you and "trustworthy" is a small number of **specific, fixable holes**: two anonymous-readable classroom tables, an anonymous write/delete path into learning data, a non-idempotent Stripe webhook, and long-lived refresh tokens in `localStorage`. None require a rewrite.

## Severity-ranked findings

### CRITICAL — fix before onboarding real schools

Nothing rises to "drop everything," but the two HIGH items below are the closest and should block a paid/school launch.

### HIGH

**H1. Any anonymous visitor can read every active classroom's roster and scores.**
`supabase/migrations/0011_rls_perf_and_indexes.sql:529-534` (and `:251-256`, `:344-349`). The `sessions`, `round_scores`, and `session_students` SELECT policies include the clause `status <> 'ended'`, with no `to authenticated` restriction. That third condition lets **anyone, including unauthenticated users**, read every non-ended session across all schools — session metadata, student rosters, and round scores. This was almost certainly meant to support "join class by code," but it leaks live classroom data tenant-wide. Fix: scope the active-session read to the specific session code the caller actually holds, and expose only the minimal columns needed to join.

**H2. `lios_state` RLS may be off entirely.**
`supabase/migrations/0005_rls.sql:190-208` only enables RLS and policies on `lios_state` *if the table existed when 0005 ran* (a conditional `do $body$ ... if exists ... $body$` block). If the table was created later, it has **RLS disabled = world-readable/writable**. This table holds per-child learning state. **Verify in the live DB** that `lios_state` has `rowsecurity = true` and both policies exist; repair if not. (Same caveat applies to any table created in the missing migrations 0001–0003 — `teachers`, `analytics_events`, `client_errors`, `skill_state*`. Run Supabase's advisor / check `pg_tables.rowsecurity` for all of them.)

**H3. Stripe webhook is not idempotent and not order-safe.**
`supabase/functions/stripe-webhook/index.ts:87-92, 121-162`. `billing_events.stripe_event_id` is `UNIQUE` (`0004:221`) but the webhook never uses it as a guard — the insert result is discarded, so a replayed event (Stripe delivers at-least-once and retries) re-applies the subscription mutation. Worse, the upserts blindly write whatever the current event says with no timestamp comparison, so an out-of-order `past_due` arriving after an `active` reverts a paying customer to locked-out (or vice-versa). Fix: short-circuit and return 200 on the unique-constraint conflict; gate mutations on a monotonic check (only apply if the event is newer than the stored row).

**H4. Long-lived refresh tokens in `localStorage` + deprecated OAuth implicit flow.**
`src/lib/supabase.ts:51-58, 220, 666-669`. The session (including the **refresh token**) is persisted to `localStorage`, and Google sign-in uses the implicit flow (`flow_type=implicit`), which returns tokens in the URL hash. Any XSS on the origin yields persistent, renewable account takeover — and these accounts can drive the authenticated Stripe checkout/portal functions. For a children's platform this is the highest-leverage auth hardening available. Fix: move to PKCE; stop persisting the refresh token in JS-readable storage (memory + httpOnly cookie).

### MEDIUM

**M1. Anonymous users can DELETE/overwrite school learning data.**
`supabase/migrations/0012_learning_attempts_policy_consolidation.sql:31-54`. The INSERT/UPDATE/**DELETE** policies grant `anon` access to any `learning_attempts` row where `child_profile_id IS NULL`. That predicate was meant for the anonymous free-play path, but **school learners also have `child_profile_id IS NULL`** (they key off `learner_id`). So any anonymous internet user can delete or tamper with school/classroom learning attempts. The read side was later locked down (`0013:460-465`) but the write side from 0012 was never re-narrowed. Fix: gate anon writes by session/tenant context, not the blanket `child_profile_id IS NULL`.

**M2. Webhook trusts mutable Stripe customer metadata for identity.**
`supabase/functions/stripe-webhook/index.ts:63-82`. Parent resolution falls back to reading `parent_id` from the Stripe **customer** object's metadata. If that metadata is ever influenced (mis-scoped key, portal config), a valid-signature event could attach subscription state to the wrong parent. Resolve identity only from your own DB (`parent_profiles.stripe_customer_id`) or metadata you set at creation time.

**M3. `billing_events.payload` stores full Stripe payloads indefinitely.**
`supabase/migrations/0004_parent_subscriptions.sql:218-225`. Raw Stripe events include adult PII (email, name, billing address). RLS correctly limits reads to the owning parent, but there's no retention/TTL and no minimization. Under GDPR data-minimization, trim to the fields you use and add a retention policy.

**M4. `tenant_id` exists on 36 tables but is used by zero RLS policies.**
`supabase/migrations/0013_tenant_isolation.sql:218-435`. Isolation works today only because every policy independently checks `parent_id`/`teacher_id`/`auth_owns_child()`. The tenant machinery is dead weight that creates a false sense of "tenant isolation" — any future table added with only a `tenant_id` assumption will be unprotected. Either wire `tenant_id` into policies or document it as informational only.

**M5. Anon-revoke is point-in-time; future RPCs default to PUBLIC EXECUTE.**
`supabase/migrations/0010_revoke_anon_execute_internal_rpcs.sql` revokes anon EXECUTE on then-existing definer functions in a one-time loop. Postgres grants EXECUTE to PUBLIC on every new function, so any future `SECURITY DEFINER` function that forgets an explicit revoke becomes anon-callable. Fix: `ALTER DEFAULT PRIVILEGES ... REVOKE EXECUTE ON FUNCTIONS FROM PUBLIC, anon`.

**M6. `analytics_events` has no rate limit, no retention, and possibly no hot-path indexes.** (See scalability section — this is the #1 thing to verify.)

### LOW

- **L1.** `VITE_ADMIN_PIN=admin1…` was committed in the initial commit (`9985f96`) and remains in git history forever. It's now **defunct** — the PIN admin gate was removed (`src/main.tsx:15-19`) and nothing in `src/` references it. Still: purge it from the env file (it's dead config) and treat that value as burned. No Stripe/anon/service-role key was ever committed.
- **L2.** `log_security_event` is anon-callable with no rate limit (log-flooding/DoS) and scrubs PII by key name only — a child nickname under an innocuous key (`"note"`) isn't stripped. `0020:57-95`.
- **L3.** CORS falls back to a real origin instead of denying unknown origins (`_shared/cors.ts:13-22`); cron-key compare isn't constant-time (`email-dispatch/index.ts:112-115`). Both minor.
- **L4.** `playlists_insert_pro` WITH CHECK doesn't pin `teacher_id = auth.uid()` (`0011:221-224`) — a pro user can attribute a playlist to another teacher.
- **L5.** Account-deletion requests rely on a processor cron that isn't scheduled in any tracked migration (`0021:151-183`); confirm it's actually wired up, or erasure requests sit forever.

### Clean / verified-good (worth stating for trust)

SQL injection: **none** — every `EXECUTE format()` uses `%I`/literals/`USING` parameters. Parent-side tenant isolation is solid. All `SECURITY DEFINER` functions have locked `search_path` and validate the caller. Consent + deletion + export flows exist end-to-end. Stripe webhook signature verification is correct (raw body via `constructEventAsync`). Checkout/portal/sync have no IDOR — customer IDs are derived server-side from the verified JWT. Service-role key is server-only.

## Will it hold? (10 → 1,000,000 users)

Two architecture choices push the breaking point far out: **clients never open direct Postgres connections** (everything goes through PostgREST, so the classic Supabase connection-exhaustion failure doesn't apply to the app tier), and **hand-tracking is 100% client-side** (per-user compute never touches a server). The kid-facing app — camera, drawing, class join — holds at every tier. What breaks first is back-office analytics and the email batch job.

| Users | Verdict | First thing that bites |
|---|---|---|
| **10** | ✅ Holds | Nothing. |
| **100** | ✅ Holds | Nothing material. |
| **1,000** | ✅ Holds (watch telemetry) | `analytics_events` write rate starts to matter — fine *if* the hot-path indexes exist. |
| **10,000** | ⚠️ Degrades | Admin insights dashboards slow (seq-scans over an unpartitioned, no-retention `analytics_events`); email-cron runtime creeps up. Users unaffected. |
| **100,000** | ❌ Breaks (operational) | `email-dispatch` times out / Resend throttles; telemetry table bloats; dashboards unusable. Kid/parent UX still OK. |
| **1,000,000** | ❌ Breaks | Telemetry ingest volume + cost; unthrottled anon inserts; storage. Needs partitioning, retention, a queue/warehouse, and server-side rate limiting. |

**The three scaling fixes, in the order they bite:**

1. **Verify/add `analytics_events` hot-path indexes** (`(occurred_at DESC)`, `(event_name, occurred_at DESC)`). `docs/ANALYTICS_PLAN.md:96-99` specifies them, but **no `CREATE INDEX` for analytics_events exists in any tracked migration** — they were applied out-of-band, so their live state is unverified. With them, degradation starts ~10k; without, low thousands. Highest impact, lowest effort.
2. **Fix `email-dispatch`** (`supabase/functions/email-dispatch/index.ts`, ~line 140): it fetches **all** trialing subscriptions with no `.limit()`/pagination, then sends emails one-at-a-time in a serial `await` loop. Paginate and cap concurrency. Bites ~10k–100k.
3. **Add retention + partitioning** on `analytics_events`/`learning_attempts` (monthly partitions + the 12-month delete cron `ANALYTICS_PLAN.md:186` already calls for but was never built), and **re-introduce server-side telemetry rate limiting** — the planned "100 events/session/min" ingest throttle (`ANALYTICS_PLAN.md:106`) was never implemented; the client writes direct to PostgREST with the public anon key (`src/lib/analytics.ts:779-855`), which is both a cost and an abuse vector at 1M.

### Bugs / broken edge cases

- **Camera resource leak (HIGH for a kids' camera app):** `src/core/useWebcam.ts:77-88`. The cleanup closure captures `stream` from the initial render, but `stream` is set asynchronously via `setStream` inside `startWebcam`, and deps are `[facingMode, autoStart]` with exhaustive-deps disabled. On unmount the closure's `stream` is often stale/null, so tracks never `.stop()` — **the webcam light can stay on after navigating away.** The production path (`useCameraController`) does this correctly via `streamRef`; the two Demo pages don't.
- **Swallowed errors:** only 2 empty catches, both low-risk (`lib/parent/parentReport.ts:263-264` around print/focus). Add a debug log.
- **Division-by-length:** product paths are guarded; the only unguarded spots are debug/measurement (`core/perf.ts:189` can yield `NaN` FPS during the startup probe). Low risk.

## Tests you need before you ship

You currently have SQL smoke tests (`supabase/migrations/tests/parent_rls_test.sql`) and a tracking smoke test, but the highest-risk paths are untested. Priority order:

**1. RLS / authorization tests (highest priority — these encode trust).** Extend the `parent_rls_test.sql` pattern to assert the *negative* cases:
- An anonymous role **cannot** read `sessions`/`round_scores`/`session_students` for another tenant (this test would have caught H1 and will catch its regression).
- An anonymous role **cannot** INSERT/UPDATE/DELETE `learning_attempts` belonging to a school learner (catches M1).
- A parent in tenant A **cannot** read/write any row of a child in tenant B.
- `lios_state` rejects cross-child reads (catches H2).
- A non-admin calling each `dashboard_*` / `lios_*` RPC gets `42501`, not data.

**2. Stripe webhook tests.**
- Replaying the same `event.id` applies the mutation **exactly once** (catches H3a).
- An out-of-order `past_due`-then-`active` (and the reverse) leaves the account in the correct final state (catches H3b).
- An event whose customer metadata points at a different `parent_id` does **not** mutate that parent (catches M2).
- A tampered-body / bad-signature request returns 400 and writes nothing.

**3. Auth-flow tests.** Session refresh works after expiry; `returnTo` rejects external URLs (open-redirect); signing out clears all persisted tokens. Add a regression test for the eventual PKCE migration (H4).

**4. Camera lifecycle test.** Mounting then unmounting a camera page stops **all** MediaStream tracks (catches the `useWebcam` leak). This is both a correctness and a privacy test for a children's product.

**5. Edge-case unit tests.** Empty-array guards in scoring/FPS math; analytics flush behavior on network failure (does it drop, retry, or block?); class-join with an invalid/expired session code.

You're closer to "trustworthy" than the length of this list suggests — the architecture is sound and the privacy model is genuinely thoughtful. The work here is closing specific doors, not rebuilding the house.
